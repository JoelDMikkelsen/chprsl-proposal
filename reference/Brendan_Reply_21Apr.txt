3. SwiftPOS CSV instead of API: temporary.

   Yes, Phase 1 only. Rationale:

   - Phase 1 ingests summary journals into NetSuite (EOD or
     intraday), which fits CSV cleanly.
   - Phase 2 introduces the middleware and service bus pattern
     Greg Chompff (our Integration Lead) walked through in the
     Integration and Data session. That throttles high-volume
     POS traffic and opens the door to richer real-time ingestion.
   - CSV in Phase 1 also removes an API build dependency from
     the 1 July go-live, so we're not chasing a vendor API
     contract inside the same window as configuration and UAT.

PHASE 2
=======

5. Supy vs Restoke: timing and Fusion5 role.

   Two quite different profiles:

   - Supy: the NetSuite connection already exists, so the work
     is configuration, data mapping and UAT. Weeks, not months.
     Fusion5's role is solution design, configuration, test and
     handover.

   - Restoke: no existing connection. We'd build one. Typical
     custom API integration of this shape runs four to eight
     weeks of build and test once the spec is agreed, assuming
     Restoke's API is documented and stable. Fusion5's role
     would be the full lifecycle: integration design, build
     (either direct API or via middleware), test, deploy and
     handover.

   Happy to look at Restoke's API documentation and come back
   with a tighter estimate if that would help. Supy is
   materially faster and cheaper to integrate, so unless
   Restoke wins on functional grounds, the cost and time delta
   is worth weighing.
